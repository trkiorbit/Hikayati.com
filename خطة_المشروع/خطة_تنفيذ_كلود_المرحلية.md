## كيف تعمل هذه الخطة

كل phase تتحوّل لـ **batches تنفيذية صغيرة**. لكل batch:
1. **اسم واضح** (مثل: `batch-002-redeem-card-ui`)
2. **phase parent** (مثل: Phase 2)
3. **نطاق محدد** (ملفات + وظائف + UI)
4. **ممنوع اللمس** (حماية)
5. **Acceptance Criteria** (قابل للتحقق)
6. **Verification** (flutter analyze + build + manual test)
7. **Commit message template**

عند انتهاء كل Phase (مجموع batches) → **نقطة استعادة**.

---

## قواعد صارمة لكل batch

### لا تفعل
- ❌ لا تضف Features خارج نطاق الـ batch
- ❌ لا تعدّل ملفات خارج القائمة المحددة
- ❌ لا تستخدم Dummy Data
- ❌ لا تغيّر Architecture
- ❌ لا تكسر قاعدة "UI عرض فقط، AI في UseCases"
- ❌ لا تلمس Auth/Session/Splash إلا في Phase 1 المخصصة لذلك
- ❌ لا تبدأ batch بدون موافقة صريحة

### افعل
- ✅ اقرأ الملفات ذات الصلة قبل التعديل
- ✅ اطلب read على كل ملف قبل write
- ✅ شغّل `flutter analyze` بعد كل تعديل
- ✅ شغّل `flutter build apk --debug` قبل إعلان "تم"
- ✅ وثّق التعديل في `سجل_التعديلات.md`
- ✅ اذكر ما لم تستطع إنجازه بدل إخفائه

## Workflow مقترح لكل batch

```
┌─────────────────────────────────────┐
│ 1. مستخدم يوافق على batch محدد    │
│ 2. كلود يقرأ الملفات ذات الصلة   │
│ 3. كلود ينفذ التعديل في النطاق   │
│ 4. flutter analyze                  │
│ 5. flutter build apk --debug        │
│ 6. تحديث سجل_التعديلات.md          │
│ 7. commit مع message واضح          │
│ 8. دفع للـ remote                  │
│ 9. طلب اختبار يدوي من المستخدم    │
│ 10. عند نجاح Phase كاملة → tag     │
└─────────────────────────────────────┘
```
