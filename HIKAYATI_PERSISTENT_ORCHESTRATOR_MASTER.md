# HIKAYATI_PERSISTENT_ORCHESTRATOR_MASTER.md

## الغرض
هذا الملف هو المرجع التنفيذي الدائم لمدير الوكلاء داخل المشروع.
وظيفته:
- إدارة المشروع على مراحل ثابتة
- تقسيم العمل على الوكلاء
- تقليل استهلاك التوكن
- حفظ الحالة الحالية دائمًا
- تمكين الاستئناف من آخر نقطة توقف بدون إعادة شرح من المستخدم
- منع الانحراف المعماري أو التوسع العشوائي

---

## القاعدة العليا
اعتبر الملفات التالية هي مصدر الحقيقة بالترتيب:

1. `HIKAYATI_AGENT_MASTER_SPEC.md`
2. `HIKAYATI_ORCHESTRATOR_PROTOCOL.md`
3. `stabilization_report.md`
4. `PHASE_A_RUNTIME_FIX_PLAN.md`
5. `HIKAYATI_RUNTIME_STATUS.md`
6. `task.md`
7. `سجل_الأخطاء.md`

إذا تعارض أي تنفيذ مع هذه الملفات، فالتنفيذ يُرفض ويجب الرجوع للمرجع الأعلى.

---

## الهدف التشغيلي
مدير الوكلاء لا يعمل كمبرمج منفرد، بل كمنسق صارم:
- يحدد المرحلة الحالية
- يحدد ما الذي يُسمح به وما الذي يُمنع
- يوزع العمل على الوكلاء
- يمنع لمس ملفات خارج نطاق المهمة
- يطلب تحققًا بعد كل دفعة عمل
- يحدث ملفات الحالة قبل التوقف
- يستأنف من آخر حالة معتمدة دون سؤال المستخدم عن ما سبق

---

## قاعدة تقليل التوكن
### ممنوع
- إرسال المشروع كاملًا للوكلاء في كل مرة
- إعادة شرح الرؤية في كل جولة
- فتح مراحل كثيرة دفعة واحدة
- طلب تقارير طويلة بعد كل تعديل صغير
- إعطاء كل وكيل سياقًا أكبر من حاجته

### المطلوب
العمل يكون على شكل **Batches** صغيرة ومحكومة:

#### كل Batch يجب أن تحتوي فقط على:
1. المرحلة الحالية
2. الهدف الجزئي
3. الملفات المسموح لمسها
4. معيار النجاح
5. ما الذي يجب التحقق منه
6. أين تُسجل النتيجة

---

## قاعدة التنفيذ الاقتصادية
استخدم هذا النمط دائمًا:

### Step 1 — Plan Brief
مدير الوكلاء يكتب باختصار شديد:
- Current Phase
- Current Batch
- الهدف
- الوكلاء المخصصون
- الملفات المسموح لمسها
- اختبار القبول

### Step 2 — Execute
يعطي الوكلاء أوامر دقيقة جدًا ومحدودة الملفات.

### Step 3 — Verify
بعد التنفيذ، لا يبدأ Batch جديد حتى يتم:
- فحص build/analyze
- فحص runtime المطلوب
- تحديث ملفات الحالة

### Step 4 — Commit State
قبل أي توقف:
- تحديث `HIKAYATI_RUNTIME_STATUS.md`
- تحديث `task.md`
- تحديث `سجل_الأخطاء.md` إذا لزم
- كتابة `NEXT_BATCH` بوضوح

---

## قاعدة الاستئناف
عند العودة في يوم لاحق:
لا يُعاد شرح المشروع.
يجب على مدير الوكلاء تنفيذ الآتي مباشرة:

1. قراءة ملفات الحالة:
   - `HIKAYATI_RUNTIME_STATUS.md`
   - `task.md`
   - `سجل_الأخطاء.md`
2. استخراج:
   - آخر Phase مكتملة
   - آخر Batch مكتملة
   - الـ Batch الجارية إن كانت متوقفة
   - المشاكل المفتوحة
   - الملفات التي كانت تحت التعديل
3. المتابعة من:
   - `NEXT_BATCH`
   - أو `CURRENT_BLOCKER`
4. عدم سؤال المستخدم عن السياق السابق إلا إذا كانت هناك معلومة مفقودة فعليًا من الملفات

## القاعدة المعمارية
لا يُسمح لمدير الوكلاء أو أي وكيل بكسر هذه القواعد:

1. الـ UI لا تنفذ orchestration مباشر
2. `CinemaScreen` شاشة عرض فقط
3. `IntroCinematicScreen` شاشة انتقال/تحميل فقط
4. الأفاتار ليس جزءًا بنيويًا من Story Wizard
5. حكيم chat منفصل عن avatar vision
6. أي feature خارج المرحلة الحالية ممنوعة
7. أي migration يجب أن تكون موثقة
8. أي إصلاح runtime لا يتحول إلى توسع feature

## سياسة تقسيم العمل على الوكلاء
### متى يستخدم أكثر من وكيل؟
استخدم عدة وكلاء فقط إذا كانت المهام:
- مستقلة
- على ملفات منفصلة
- لها تحقق مستقل

### متى يستخدم وكيل واحد؟
إذا كانت المهمة:
- صغيرة
- على ملفين أو ثلاثة
- فيها احتمال تعارض مرتفع
- تتطلب تتبعًا خطيًا

### القاعدة
لا تستخدم أكثر من **3 وكلاء** في دفعة واحدة إلا لو كانت المهمة كبيرة ومقسمة بوضوح.

### ممنوع على الوكلاء
- تلخيص المشروع كله
- إعادة سرد الـ spec كاملًا
- قراءة ملفات غير لازمة
- فحص المشروع كاملًا في كل دفعة

### استخدم هذا المبدأ
**Read narrow → Change narrow → Verify narrow**

## تسلسل التشغيل القياسي
مدير الوكلاء يجب أن يعمل بهذه الدورة دائمًا:

### A. Read State
اقرأ فقط ملفات الحالة والملف المرجعي اللازم للمهمة

### B. Define Batch
حدد:
- الهدف
- الوكلاء
- الملفات
- التحقق

### C. Execute Batch
نفّذ التعديلات

### D. Verify Batch
نفّذ:
- `flutter analyze --no-pub`
- اختبار runtime المحدد فقط لهذه الدفعة

### E. Update State
حدّث ملفات الحالة

### F. Stop Cleanly
إذا انتهى الوقت أو التوكن:
- لا تبدأ دفعة جديدة
- اكتب `NEXT_BATCH`
- توقف

---

## سياسة التحقق
لا يُقبل أي Batch بدون تحقق مناسب.

### التحقق الأدنى
- analyze pass أو لا توجد errors حقيقية
- اختبار المسار المتأثر
- التأكد أن التعديل لم يكسر route أو service أو model

### التحقق الموسع
استخدمه فقط إذا كانت الدفعة حساسة:
- runtime logs
- DB verification
- API response verification

---

## سياسة المراحل
لا ينتقل مدير الوكلاء إلى مرحلة جديدة إلا إذا:
1. المرحلة الحالية أغلقت رسميًا
2. ملفات الحالة محدثة
3. التحقق النهائي مكتمل
4. لا توجد blockers حرجة مفتوحة
5. المستخدم طلب الانتقال أو الملف المرجعي يفرض الاستمرار الآمن

---

## المراحل المعتمدة للمشروع
### Phase A — Runtime Stabilization
حل مشاكل التشغيل الحالية فقط

### Phase 1 — Story Flow Foundation
إنشاء القصة → المقدمة → السينما

### Phase 2 — Auth + Save + Private Library
توثيق المستخدم + حفظ القصة + مكتبة خاصة

### Phase 3 — Public Library + Credits
المكتبة العامة + الفتح بالكريدت

### Phase 4 — Settings + Policies + Standard UX hardening

### Phase 5 — Store shell + payments integration

### Phase 6 — Avatar advanced layer + voice clone

### Phase 7 — Hakeem advanced + video + school mode

---

## قاعدة الانتقال بين المراحل
كل مرحلة تُنفذ عبر عدة Batches.
مثال:

### Phase 2
- Batch 2.1 — Auth routing
- Batch 2.2 — session handling
- Batch 2.3 — save story repository
- Batch 2.4 — private library fetch/render
- Batch 2.5 — verification + cleanup

لا يتم فتح Batch 2.2 قبل نجاح 2.1.

## إذا ظهرت مشكلة جديدة
إذا ظهر blocker جديد:
1. سجله في `HIKAYATI_RUNTIME_STATUS.md`
2. سجله في `سجل_الأخطاء.md`
3. قيّم هل يمنع متابعة الـ Batch الحالية
4. إذا كان blocker حرج:
   - أوقف الانتقال
   - عرّف Mini Batch خاصة بحله
5. ثم واصل

---

## إذا انتهى التوكن أو الوقت
قبل التوقف يجب إلزاميًا:
1. تحديث `HIKAYATI_RUNTIME_STATUS.md`
2. تحديث `task.md`
3. كتابة `NEXT_BATCH`
4. كتابة سطر واحد يوضح:
   - أين توقف التنفيذ
   - ما الذي تبقى
   - ما الاختبار القادم
